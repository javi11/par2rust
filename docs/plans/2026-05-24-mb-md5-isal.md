# Multi-Buffer MD5 via ISA-L FFI — Implementation Plan

> Companion to [`docs/plans/2026-05-22-gfni-kernel.md`](2026-05-22-gfni-kernel.md). Targets gap #3 from the original ParPar comparison: per-stream MD5 throughput, which is the scan-phase bottleneck on multi-core hosts now that the GFNI encoder lands ~20 GiB/s.

## Context

`benches/md5_scan.rs` (PR #17) measured Apple M4 at:

| Backend           | Serial     | Rayon (10 cores) |
|-------------------|------------|------------------|
| `md-5` (default)  | 720 MiB/s  | 4.95 GiB/s       |
| `openssl` (asm)   | 813 MiB/s  | 5.46 GiB/s       |

Encoder kernel (NEON GFNI, single-thread): ~20 GiB/s. **MD5 is ~4× slower per stream than encode**, so scan is the bottleneck on multi-core. ParPar's `md5mb-neon` lanes 4 independent streams in 1 NEON register and reports ~3 GiB/s per stream. ISA-L `md5_mb_asimd_x4` is the equivalent design (4-way) with a managed lane scheduler. Either path should push aggregate MD5 from ~5 GiB/s to ~20 GiB/s on M4 — closing the gap to the encoder.

This plan picks **ISA-L FFI** (battle-tested C from Intel, BSD-3-Clause, used by Linux kernel / MinIO / etc) over hand-porting the SIMD assembly. Trade-off: build complexity vs SIMD correctness risk.

## Scope

**In scope (Phase 1):**
- Vendor the minimal set of `isa-l_crypto/md5_mb` files needed for aarch64 + x86_64 multi-buffer MD5.
- `build.rs` that compiles them via `cc::Build`, dispatching aarch64 / x86 file lists by target.
- Raw FFI bindings (hand-written, not bindgen — the API surface is ~5 functions) in a new `src/md5_mb.rs` module.
- Standalone bench `benches/mb_md5_isal.rs` proving correctness (digests match `md-5`) and measuring throughput vs the baseline `md5_scan` bench.

**In scope (Phase 2, separate PR):**
- Plumb a `MultiBufferMd5` lane manager through `scan_via_mmap`'s `par_chunks(slice_size)` so per-slice MD5 runs through ISA-L lanes within each rayon worker.
- Cargo feature `mb-md5` (default off; users opt-in once perf is verified across hosts).

**Out of scope:**
- SVE / SVE2 implementations (Apple Silicon doesn't expose them; Graviton / Neoverse follow-up).
- Replacing the `hash_full` and `hash16k` paths (single streams, multi-buffer doesn't help).
- Cross-architecture autotools `configure` — we hand-pick the file list per target and build directly.

## Tech stack

- **C library**: [`intel/isa-l_crypto`](https://github.com/intel/isa-l_crypto), BSD-3-Clause. Vendored at a pinned commit (NOT a submodule — vendored to keep `cargo publish` self-contained).
- **Build**: `cc::Build` from `build.rs`. No autotools, no nasm-on-x86 dependency for aarch64-only builds.
- **FFI**: hand-written Rust. ~5 functions, 1 opaque manager type, 1 job struct.

## API surface (FFI)

ISA-L's public `multi_buffer.h` exposes (simplified):

```c
typedef struct MD5_HASH_CTX_MGR MD5_HASH_CTX_MGR;
typedef struct MD5_HASH_CTX     MD5_HASH_CTX;

void md5_ctx_mgr_init(MD5_HASH_CTX_MGR *mgr);
MD5_HASH_CTX* md5_ctx_mgr_submit(MD5_HASH_CTX_MGR *mgr, MD5_HASH_CTX *ctx,
                                 const void *buffer, uint32_t len, HASH_CTX_FLAG flags);
MD5_HASH_CTX* md5_ctx_mgr_flush(MD5_HASH_CTX_MGR *mgr);
```

`HASH_CTX_FLAG` is `HASH_FIRST | HASH_UPDATE | HASH_LAST | HASH_ENTIRE`. For per-slice PAR2 MD5 we use `HASH_ENTIRE` (single submit per slice).

Rust wrapper (sketch in `src/md5_mb.rs`):

```rust
use std::cell::UnsafeCell;
use std::os::raw::{c_uint, c_void};

#[repr(C)]
pub struct Md5MbMgr {
    _opaque: [u8; ISA_L_MGR_SIZE],
}

#[repr(C)]
pub struct Md5MbCtx {
    _opaque: [u8; ISA_L_CTX_SIZE],
}

extern "C" {
    fn md5_ctx_mgr_init(mgr: *mut Md5MbMgr);
    fn md5_ctx_mgr_submit(mgr: *mut Md5MbMgr, ctx: *mut Md5MbCtx,
                          buffer: *const c_void, len: u32, flags: c_uint) -> *mut Md5MbCtx;
    fn md5_ctx_mgr_flush(mgr: *mut Md5MbMgr) -> *mut Md5MbCtx;
}

pub struct Md5MultiBuffer { /* ... */ }

impl Md5MultiBuffer {
    pub fn new(capacity: usize) -> Self { /* ... */ }
    pub fn digest_all(&mut self, slices: &[&[u8]]) -> Vec<[u8; 16]> {
        // 1. Allocate ctxs (one per slice).
        // 2. Submit each with HASH_ENTIRE; a return of non-null means
        //    that lane completed and we pull its digest.
        // 3. Flush remaining lanes; pull digests in completion order.
        // 4. Reorder digests back to input order via stable index tag.
    }
}
```

The trickiest detail: `submit` may return a completed CTX *other than the one just submitted* (the lane scheduler is non-FIFO). We track each ctx's slot via a small lookup so digests come out in the right order.

`ISA_L_MGR_SIZE` and `ISA_L_CTX_SIZE` come from looking at the C struct definitions in `md5_mb_mgr_datastruct.h` and `md5_job.h`. We hard-code the sizes (with a `build.rs` compile-time assertion via `static_assertions::const_assert_eq!(...)` against ISA-L's headers) so the Rust side can stack-allocate without a heap dance.

## Tasks (executable, in order)

> Mark each with `- [ ]` then `- [x]` as the implementer subagent works.

### Task 1: Vendor the ISA-L sources

- [ ] Pick a pinned commit SHA from `isa-l_crypto` master (latest stable at time of execution, e.g. v2.25.0).
- [ ] Download a tarball at that SHA: `gh api repos/intel/isa-l_crypto/tarball/<SHA>`.
- [ ] Extract just `md5_mb/`, `include/multi_buffer.h`, `include/types.h`, `include/datastruct.asm`, and `LICENSE` into `vendor/isa-l_crypto/`.
- [ ] Add a `vendor/isa-l_crypto/README.md` recording the upstream SHA + extraction date + GPL-compatibility note (ISA-L is BSD-3-Clause).
- [ ] Verify the vendored tree builds standalone on aarch64 macOS: `cd vendor/isa-l_crypto && cc -c md5_mb/md5_ctx_base.c -I include -o /tmp/test.o`.
- [ ] **Commit**: `chore(vendor): bring in isa-l_crypto md5_mb at <SHA> for multi-buffer MD5`.

### Task 2: build.rs

- [ ] Add `cc = "1"` to `[build-dependencies]` in `Cargo.toml`.
- [ ] Add a `mb-md5` feature (off by default): `mb-md5 = []`. Gate everything below on this feature so a default build is unchanged.
- [ ] Create `build.rs` (or extend an existing one):
  - Detect target arch.
  - On aarch64: compile the aarch64 file list (`.c` + `.S`).
  - On x86_64: detect `nasm` via `which`. If found, compile full asm path. If not, emit a `cargo:warning` and compile only the base C path.
  - Other archs: emit `cargo:warning` and disable the feature at compile time via a `cfg`.
  - Link as a static lib named `isal_md5_mb`.
- [ ] **Commit**: `feat(build): vendor isa-l_crypto md5_mb via cc::Build (gated on mb-md5 feature)`.

### Task 3: FFI bindings + safe wrapper

- [ ] Create `src/md5_mb.rs`. Module is feature-gated on `mb-md5`.
- [ ] Define `extern "C"` block matching `multi_buffer.h`.
- [ ] Define `Md5MultiBuffer` safe wrapper that:
  - Owns a heap-allocated `Md5MbMgr` (don't stack-allocate — ISA-L manager is large).
  - Owns a `Vec<Md5MbCtx>` for in-flight jobs.
  - Maps submitted lanes to caller indices via a `Vec<usize>` slot table.
  - Exposes `digest_all(slices: &[&[u8]]) -> Vec<[u8; 16]>`.
- [ ] Wire into `crate::md5_impl` only when `mb-md5` is enabled — but DO NOT change `scan_via_mmap` yet (that's Phase 2). For now, just expose the multi-buffer API.
- [ ] **Commit**: `feat(md5_mb): safe Rust wrapper around isa-l_crypto multi-buffer MD5`.

### Task 4: Correctness test (CRITICAL)

- [ ] In `src/md5_mb.rs::tests`, add an exhaustive test:
  - Generate N (= 1, 4, 5, 16, 17, 64, 100, 200) deterministic-pseudorandom slices of varying lengths (1 byte, 63 bytes, 1 MiB, 1 MiB + 1, 16 MiB).
  - Run via `Md5MultiBuffer::digest_all`.
  - Run via `md_impl::digest` (scalar) per slice.
  - Assert digests match byte-for-byte across all inputs.
- [ ] If any mismatch: the slot-table ordering is wrong, OR ISA-L's submit/flush state machine isn't being driven correctly. Bisect by N=1 (single lane), then N=4 (exactly fills lanes once).
- [ ] **Commit**: `test(md5_mb): cross-validate ISA-L multi-buffer digests against md-5 (RFC 1321)`.

### Task 5: Bench

- [ ] Create `benches/mb_md5_isal.rs`. Feature-gated on `mb-md5`.
- [ ] Same workload as `md5_scan.rs` (200 × 1 MiB), but with an extra `multibuffer` group using `Md5MultiBuffer::digest_all`.
- [ ] Record M4 numbers in this plan file under "Phase 1 results".
- [ ] **Commit**: `bench(mb_md5): criterion harness comparing ISA-L mb_md5 vs scalar/openssl backends`.

### Task 6: Phase 1 PR

- [ ] Open PR against `main` summarising:
  - Vendored ISA-L SHA + license
  - Per-target build behaviour (when feature on / off)
  - Apple M4 measured numbers (filled in by Task 5)
  - Explicit "Phase 2 (scan integration) is NOT in this PR"
- [ ] **Do NOT merge** until Apple M4 throughput is ≥ 2× the rayon baseline. If less, abandon and document why in `vendor/isa-l_crypto/README.md`.

## Risks and mitigation

| Risk | Likelihood | Mitigation |
|---|---|---|
| ISA-L assembler symbols clash with macOS `_`-prefix convention | medium | `cc::Build` handles symbol decoration. If linker errors, check `--target` flag and add `cargo:rustc-link-arg`. |
| `nasm` not available in CI | low (Linux/macOS have it via apt/brew) | aarch64 path doesn't need it; on x86 fall back to C base + emit `cargo:warning`. |
| ISA-L API uses non-FIFO completion order; off-by-one in slot table → wrong digests | **high** | Task 4 test must be exhaustive. Run N=1, 4, 5 in isolation when bisecting. |
| ISA-L struct sizes change between versions, breaking our hard-coded sizes | low (pinned commit) | `static_assertions::const_assert_eq!(size_of::<Md5MbMgr>(), ...)` against the C struct via a small build.rs assertion. |
| GPL-2.0-or-later + BSD-3-Clause license combine cleanly | known | Both compatible. Output is GPL. Document in vendor README. |
| ASIMD x4 doesn't actually win on Apple Silicon | medium-low | Bench in Task 5 is the gate. If ≤ 1.5× baseline, abandon and document. |

> Filled in during Task 5 execution.

| Backend           | Serial      | Rayon (10 cores) | Multi-buffer (1 thread, 4 lanes) |
|-------------------|-------------|------------------|-----------------------------------|
| `md-5` (default)  | 720 MiB/s   | 4.95 GiB/s       | _TBD_                             |
| `openssl`         | 813 MiB/s   | 5.46 GiB/s       | _TBD_                             |
| `mb-md5` ASIMD x4 | _TBD_       | _TBD_            | _TBD_                             |

Decision gate: **proceed to Phase 2 iff Phase 1 single-thread mb-md5 ≥ 2 GiB/s** (i.e. ≥ 2.5× scalar). Below that, the integration complexity isn't justified.

## Phase 2 preview (separate PR after Phase 1 lands)

- Refactor `scan_via_mmap`'s `par_chunks(slice_size).map(|s| digest(s))` into a `par_chunks(slice_size).chunks(4).map(|batch| mb_md5.digest_all(batch))` pattern.
- Each rayon worker owns one `Md5MultiBuffer` (thread-local), reused across all its batches.
- Decision point: switch `mb-md5` to a default feature once it's proven correct + faster on at least one Linux x86 box too.
